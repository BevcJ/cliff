Last Outreach Follow-up

Add a manually maintained company-level outreach date and convert its age into an accessible table reminder.

1. Overview

1.1 Problem Statement

The inspection UI stores a current outreach status but no date for the most recent outbound message. Existing review timestamps change for unrelated edits and cannot determine when a follow-up is due.

1.2 Proposed Solution

Store a nullable last_outreach_date on each company review-state row. Operators edit Fit Status, Outreach Status, and Last Outreach directly in the company table. Company Detail remains the editor for General Notes and Communication History only.

1.3 Terminology

TermDefinition
Last OutreachCalendar date of the latest manually recorded outbound message for a company.
Reminder statusmessage_sent or follow_up_sent; statuses for which age is actionable.
Elapsed daysCurrent server calendar date minus Last Outreach.

2. Architecture

2.1 Current Flow

company_review_state.outreach_status
  -> merge_review_state()
  -> company table and read-only detail facts

2.2 Proposed Flow

company_review_state.last_outreach_date (nullable DATE)
  -> load and serialize as YYYY-MM-DD
  -> merge into generated company record
  -> table date editor for writes
  -> pure age calculation at table render
  -> pinned Follow-up indicator + editable Last Outreach column

The active grid uses server-authoritative synchronization after reruns. Status edits use upsert_review_statuses(), date edits use upsert_last_outreach_date(), and notes/history edits use upsert_review_notes(). Each path updates only its owned fields.

4. Edge Cases & Behaviour

EC-1: Missing date

A reminder status with no date displays a red "Date missing" indicator.

EC-2: Boundary days

Days 0-3 are green, days 4-5 are yellow, and day 6 onward is red.

EC-3: Inactive status

not_started, active_conversation, closed, and lost statuses show no age indicator even if a date remains stored.

EC-4: Future date

The table date editor prevents selection after today, and persistence validation rejects future dates.

EC-5: Partial saves

Status edits preserve Last Outreach and notes/history. Date edits preserve statuses and notes/history.

EC-6: Stale browser rows

The grid uses server_wins synchronization so Postgres-backed reruns replace old client row data.

6. Decisions Made

D-1: Company-level current state

Store one date per company because the existing workflow and review-state primary key are company-level. Contact-level event history is deferred.

D-2: SQL DATE

Use a calendar date rather than a timestamp because entry and reminders operate in whole days and do not require send-time precision.

D-3: Table entry

Make Fit Status, Outreach Status, and Last Outreach editable only in the table. This keeps row triage fast and avoids conflicting table/detail controls.

D-4: Text plus color

Use an icon and label together so reminder meaning is not communicated by color alone.

D-5: New status mapping

Apply age reminders to message_sent and follow_up_sent. The newer active_conversation status replaces the old replied state and suppresses reminders.

7. Upgrade Paths

7.1 Configurable thresholds

Move the named 3-day and 5-day constants into deployment configuration if different teams need different cadences.

7.2 Outreach event history

Add a contact/message event table and derive Last Outreach when recipient-level history and auditing become requirements.

7.3 Notifications

Add scheduled email or in-app notifications after the visual workflow has proven useful.
